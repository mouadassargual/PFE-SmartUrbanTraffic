\section{Construction du dataset d'entraînement}

\subsection{Problématique et objectifs}

La constitution d'un dataset adapté à notre
contexte a représenté l'un des défis majeurs
de ce travail. Notre système requiert la
détection fiable de six classes dans des
scènes de surveillance urbaine :
\textbf{bus, car, emergency\_vehicle,
motorcycle, person et truck}.

Deux contraintes spécifiques ont guidé
la construction :

\begin{itemize}
    \item \textbf{Contrainte Privacy-by-Design :}
    la classe \textit{person} doit être détectée
    avec une précision maximale, car tout piéton
    non détecté échappe à l'anonymisation
    obligatoire au titre de la Loi 09-08.

    \item \textbf{Contrainte embarquée :}
    le dataset doit produire un modèle
    suffisamment léger pour s'exécuter
    en temps réel sur Raspberry Pi 5.
\end{itemize}

\subsection{Méthodologie de construction
            en trois étapes}

La construction du dataset a suivi une
méthodologie rigoureuse en trois étapes
successives, chacune répondant à un
problème empiriquement identifié.

\subsubsection{Étape 1 — Split sans fuite
               de données}

Les images étant extraites de vidéos,
des frames proches d'une même séquence
peuvent se retrouver simultanément dans
les ensembles d'entraînement et de test.
Ce phénomène produit des métriques
artificiellement élevées, car le modèle
a vu des frames quasi-identiques pendant
l'entraînement.

Pour éviter ce biais, un split par
\textbf{groupes de source} a été appliqué :
toutes les frames extraites d'une même
vidéo appartiennent au même split.

\subsubsection{Étape 2 — Renforcement
               de la classe person}

La classe \textit{person} était sous-représentée
dans les données de surveillance (2 075 instances
en train). Or, cette classe est doublement critique :
détecter les piétons pour la décision MDP, et
les anonymiser pour la conformité Loi 09-08.

Le dataset \textbf{CityPersons} (18 villes,
piétons de tailles variées) a été intégré
\textbf{exclusivement dans l'ensemble
d'entraînement} afin de ne pas biaiser
l'évaluation sur val et test.

\subsubsection{Étape 3 — Crops pour
               tiny persons}

Dans les vidéos de surveillance d'intersections,
les piétons sont fréquemment très petits dans
l'image (< 0.1\% de la surface totale). YOLO
peut facilement ignorer ces détections.

Pour traiter ce problème, 531 crops centrés
sur les boîtes \textit{person} de très petite
taille ont été générés et ajoutés à l'ensemble
d'entraînement. Toutes les classes visibles
dans chaque crop ont été conservées, avec
recalcul des coordonnées YOLO.

\section{Benchmark et sélection du modèle
         de détection}

\subsection{Protocole expérimental}

Afin de sélectionner le modèle de détection
optimal pour notre système, un benchmark
comparatif a été conduit en deux phases
complémentaires : une phase d'entraînement
sur GPU (Google Colab Pro, NVIDIA A100)
pour évaluer la précision, suivie d'une
phase de déploiement sur Raspberry Pi 5
pour évaluer les performances en conditions
réelles.

Six architectures YOLO ont été évaluées,
couvrant deux générations de variantes
nano et small :

\subsection{Analyse des résultats}

\subsubsection{Nano vs Small —
               Validation empirique}

Le benchmark confirme empiriquement
que les variantes Small sont
\textbf{incompatibles avec le temps réel}
sur Raspberry Pi 5 : aucune variante
small n'atteint 3 FPS, quelle que soit
la précision du format d'exportation.
À l'inverse, toutes les variantes nano
dépassent le seuil de 5 FPS requis
pour une gestion adaptative des feux
de signalisation.

Ce résultat est cohérent avec les
recommandations de Wang et al. (2025),
qui identifient les architectures légères
comme la solution privilégiée pour les
dispositifs edge à ressources limitées.

\subsubsection{FP32 vs INT8}

La quantification INT8 réduit fortement
la taille du modèle, mais n'apporte pas
un gain FPS systématique sur Raspberry Pi 5
et introduit une confusion plus marquée entre
les classes visuellement proches, notamment
\textit{car}, \textit{bus} et \textit{truck}.
Ce phénomène est cohérent avec Wang et al. (2025),
qui signalent une chute de 4.7 à 6.2\% de l'AP
sur les petits objets sous quantification INT8.
Dans notre redéploiement final, l'export INT8
dynamique du modèle Step 3 800 réduit bien la taille
de l'artefact ONNX de \textbf{9.4 MB} à \textbf{2.8 MB},
mais il descend à \textbf{2.06 FPS} sur Raspberry Pi 5,
contre \textbf{3.94 FPS} pour le FP32. Le format
\textbf{FP32 a donc été retenu} pour garantir à la fois
la fiabilité des détections et le meilleur débit mesuré.

\subsubsection{Modèle final retenu —
               YOLO26n FP32}

Parmi les variantes nano en FP32,
\textbf{YOLO26n} est retenu comme
famille de modèles de déploiement pour les raisons
suivantes :

\begin{enumerate}
    \item \textbf{FPS le plus élevé :}
    7.6 FPS réels — la marge la plus
    importante au-dessus du seuil minimal,
    laissant des ressources CPU disponibles
    pour le tracking, l'anonymisation
    et le dashboard Flask.

    \item \textbf{Architecture NMS-Free :}
    YOLO26n supprime l'étape
    Non-Maximum Suppression, réduisant
    la latence sur CPU ARM~\cite{sapkota2025}.

    \item \textbf{Modèle léger :}
    environ 9.6 à 9.8 MB en ONNX FP32 —
    empreinte mémoire minimale
    sur le Raspberry Pi 5.

    \item \textbf{Fine-tuning final :}
    YOLO26n Step 3 960 améliore la mAP50
    test à 0.792 et le rappel \textit{person}
    à 0.353.
\end{enumerate}

Un redéploiement final du modèle Step 3 960 sur
Raspberry Pi 5 a ensuite été réalisé avec les ROI
personnes et véhicules activées. Sur 300 frames de
la vidéo NE8th, le pipeline complet atteint
\textbf{2.16 FPS} en moyenne, soit environ
\textbf{463 ms/frame}. Deux variantes plus légères
ont aussi été mesurées : \textbf{2.48 FPS} avec ROI
personnes toutes les 10 frames et sans ROI véhicules,
puis \textbf{2.51 FPS} sans ROI additionnelles. Le gain
reste donc limité, ce qui indique que le coût principal
vient de l'inférence ONNX en 960$\times$960 pixels.
Cette configuration est retenue comme mode d'analyse
haute précision, tandis qu'un mode temps réel fluide
nécessite un export plus léger ou une réduction de la
fréquence d'inférence. Un export expérimental
800$\times$800 depuis les mêmes poids a ensuite été
testé : il atteint \textbf{3.94 FPS} sans ROI et
\textbf{3.45 FPS} avec ROI personnes toutes les
10 frames lorsque toutes les frames sont inférées.
Le gain est significatif, mais reste inférieur au
seuil de 5 FPS en inférence exhaustive. Une dernière
optimisation logicielle, \texttt{--vid-stride 2},
infère une frame source sur deux : elle atteint
\textbf{4.28 FPS IA} et couvre \textbf{7.84 frames
source/s}. Cette valeur dépasse le seuil de 5 FPS en
couverture vidéo, mais doit être distinguée du FPS
d'inférence réel. L'INT8 dynamique, malgré sa taille
plus faible, a été rejeté car il ralentit le pipeline.

\begin{resultbox}{Modèle retenu —
                  YOLO26n FP32}
YOLO26n FP32 est sélectionné comme base
de déploiement. La variante Step 3 960
est retenue comme meilleur modèle de
précision, tandis que la variante Step 3
800 exportée depuis ces poids devient le
meilleur candidat Pi expérimental
(\textbf{3.94 FPS} sans ROI, ou
\textbf{7.84 frames source/s} avec
\texttt{--vid-stride 2}). Le stride permet
d'atteindre le seuil de couverture vidéo,
mais seulement une frame sur deux est inférée.
\end{resultbox}

projet privilégie l'exécution en module Python afin
d'éviter les erreurs d'import liées au package
\texttt{pipeline}.

caption={Commande de lancement du pipeline final},
label={lst:pipeline_command}]
python3 -m pipeline.main \
  --video data/videos/ne8th/Bellevue_Bellevue_NE8th__2017-09-11_14-08-31_3min.mp4 \
  --model models/downloads/YOLO26n_step3_960_best.onnx \
  --imgsz 960 \
  --conf 0.35 \
  --dashboard \
  --person-roi \
  --vehicle-roi

Le dashboard Flask est ensuite accessible à l'adresse
\texttt{http://<ip-raspberry>:5000}. Il permet de
sélectionner un instant vidéo, de lancer une analyse
fixe, puis de vérifier simultanément les détections,
l'anonymisation, les polygones N/E/S/W et la décision
MDP.

\subsection{Couche DataOps, MLOps et DevOps}

Afin de rendre le travail plus reproductible, une
couche de traçabilité a été ajoutée autour du
pipeline. Elle distingue trois niveaux
complémentaires.

\begin{itemize}
    \item \textbf{DataOps :} le fichier
    \texttt{dvc.yaml} décrit les étapes de
    construction du dataset : split sans fuite
    de données, ajout de CityPersons uniquement
    dans l'ensemble d'entraînement, puis génération
    des crops pour les très petits piétons. Le
    dataset final est suivi par DVC via
    \path{data/dataset/dataset_step3_tiny_person_crops.dvc}
    (\textbf{13 526 fichiers}, environ \textbf{1.28 Go}).
    Les étapes intermédiaires sont gelées
    (\textit{frozen}) afin de documenter la lignée
    sans relancer les traitements lourds.

    \item \textbf{MLOps :} le fichier
    \path{params.yaml} centralise les paramètres
    du modèle final, de l'inférence et du déploiement.
    Les métriques finales sont conservées dans
    \path{metrics/final_yolo26n_step3_960.json}.
    Le script \path{scripts/log_mlflow_final.py}
    permet de créer une exécution MLflow locale
    adossée à une base \path{mlflow.db}, contenant
    les paramètres, les métriques et les artefacts
    finaux, dont le fichier PyTorch et l'export ONNX.
    L'exécution finale porte le nom
    \texttt{YOLO26n-Step3-960-final-52b92829}.

    \item \textbf{DevOps :} la commande
    \texttt{python3 -m pipeline.main} standardise
    l'exécution du pipeline, l'export ONNX facilite
    le passage vers Raspberry Pi 5, et le dashboard
    Flask fournit une interface de validation
    opérationnelle accessible sur le port 5000.
    Le dépôt GitHub complète cette couche en
    séparant la branche \texttt{main}, réservée
    aux versions stables du projet, et la branche
    \texttt{develop}, utilisée pour intégrer les
    évolutions du pipeline, du rapport et des
    figures avant validation.
\end{itemize}

Cette couche reste volontairement légère : elle
utilise un cache DVC local, une base MLflow SQLite
et un dépôt GitHub pour la gestion du code. Les commandes
principales sont :
\path{scripts/run_dvc.sh} \texttt{status} pour vérifier
la cohérence DVC, \path{scripts/run_dvc.sh} \texttt{metrics show}
pour lire les métriques finales, puis
\texttt{python3} \path{scripts/log_mlflow_final.py}
pour mettre à jour l'exécution MLflow. Le cycle de
développement suit le schéma
\texttt{develop} $\rightarrow$ Pull Request
$\rightarrow$ \texttt{main}.
